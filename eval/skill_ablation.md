| **模型** | deepseek-chat（通过 DeepSeek API） |

## 关键差异分析

### 1. "信息可信度说明" — 最大差异

Group A 的输出末尾包含一段完整的可信度声明：

> - ✅ **已确认**：元数据来自 B 站公开 API，转写内容来自 faster-whisper ASR
> - ⚠️ **转写质量**：ASR 转写存在少量同音错字
> - ❌ **未使用**：未使用画面 OCR

Group B 完全没有这部分内容。

**原因**：Skill 的「输出格式」模板和「忠实性规则」明确要求生成此节。无 Skill 时，通用 system prompt 不包含"对内容来源标注可信度"的要求。

### 2. 输出格式规范性

Group A 严格遵循 skill 定义的 `index.md` 模板结构（来源信息 → 内容提要 → 核心要点 → 注意事项 → 可信度说明），输出适合直接作为知识库条目。

Group B 使用自由格式（🎯 📺 🔧 ⚠️ emoji 小节标题），更像一次性的对话回复，不利于后续 RAG 检索。

### 3. 基本信息完整性

Group A 的基本信息表比 Group B 多了「标题」和「链接」两个字段。这对应 skill 「输出格式」模板中「## 来源信息」的明确字段要求。

### 4. 实战案例亮点

Group A 单独列出了「实战案例亮点」小节（高考志愿填报、618 电商复盘、Go 模式），将案例从八大能力描述中抽离出来单列。Group B 将案例融入了各能力的描述段落中。

## 讨论

### 本次实验中两者均成功完成任务的原因

1. 任务属于"读取已有数据+总结"——相对简单，模型凭通用能力即可完成
2. DeepSeek-chat 模型本身具备较强的总结和格式化能力
3. 任务中明确告知了已有数据路径，Agent 不需要自主决策工具链

### Skill 真正体现价值的场景（本次实验未覆盖）

1. **处理新视频**：Agent 需要自主决定 `video_probe → video_transcribe → video_frame_ocr → kb_write` 的调用顺序和容错策略
2. **知识库落盘**：需要按固定目录结构 (`knowledge_base/<BV>/`) 生成标准化文件
3. **忠实性强制**：无 Skill 时 Agent 可能在无法获取视频内容时仍编造内容（"幻觉"风险）
4. **输出被其他系统消费**：当输出需要被 RAG 管道或自动化流程解析时，格式一致性至关重要

### 实验局限性

- 仅各运行 1 次，未取均值
- 任务场景单一（已有数据 vs 全新处理）
- 未测量 token 消耗差异（skill body 会占用更多 prompt token，但在处理新视频时可减少重试轮数）
